# 标讯情报 Skill

## 目标

把不同招投标网站中的公告转成统一、可追溯、可去重的标讯记录。默认范围是贵州省，默认行业是图文广告；区域、行业和信息源都必须配置化，不能写死在采集逻辑里。

## 安全边界

- 信息源账号和密码只允许进入 `data/private/tenders.sqlite3` 或运行环境的密钥服务。
- 不在对话、日志、日报、代码、Skill 文档中输出明文密码。
- 登录网站前先确认该账号属于用户或用户所在组织，并遵守网站规则。
- 遇到验证码、短信验证、人脸验证或协议确认时，标记为“需要人工”，不要绕过。
- 日报必须保留公告原始网址；无法核实原文的候选项进入复核队列，不直接推送为确定信息。

## 每日工作流

1. 读取 `config/regions.json` 中的当前区域。
2. 读取 `config/industries/` 中已启用行业及关键词。
3. 从数据库读取启用的信息源，优先处理无需登录的公开公告，再处理需要登录的平台。
4. 需要联网时加载 `web-access`，按照网站正常交互路径查询最近 24-48 小时公告。
5. 对每条候选公告提取统一字段，字段规范见 `references/tender-schema.md`。
6. 用公告规范化网址作为第一去重键；缺少网址时使用“项目名称 + 招标人 + 截止时间”指纹。
7. 区域明确命中排除省份时剔除；区域不明确时进入复核，不擅自判断为贵州。
8. 标题和主要内容至少命中一个行业关键词才进入行业日报；同时记录全部命中词。
9. 截止时间已过的公告不进入“今日可投”主列表，但可作为更正或结果公告保存。
10. 生成 8:00 日报：新增数量、重点项目、最早截止项目、需人工复核项、来源异常。
11. 有有效订阅授权时调用微信订阅消息；无权限时只更新小程序日报页面并记录原因。

## 信息源维护

新增或更新信息源时：

- 先更新用户维护的源表，再运行导入，不直接手改数据库。
- 一个网站可有多个公司账号，每个账号保留独立状态。
- 记录登录是否有效、最后验证时间、验证码类型、可查询地区、适配器状态。
- 网站结构变化时新增或修复对应适配器，不用脆弱的全站通用选择器替代。

## 行业维护

每个行业使用一个独立 JSON 文件，至少包含：

- `industry_id`
- `name`
- `keywords`
- `exclude_keywords`
- `match_mode`

同义词应加入关键词库。明显会产生大量误报的词应加入排除词，而不是在代码中临时判断。

## 输出要求

给用户汇报时始终分开说明：

- 已采集并核实的标讯
- 疑似相关、等待人工复核的标讯
- 信息源登录或采集异常
- 微信消息是否实际发送，还是仅更新了小程序日报

不要把“程序已接好”描述成“每天推送已上线”；只有小程序、服务端、模板、用户订阅授权和 8:00 定时任务都验证成功后，才可称为上线。

## 本地入口

初始化附件：

```bash
PYTHONPATH=src python3 -m tender_agent.cli bootstrap \
  --sources /path/to/01信息源库.xlsx \
  --keywords /path/to/02图文广告行业关键词库.txt \
  --history /path/to/03标讯信息表.xlsx
```

查看数据状态：

```bash
PYTHONPATH=src python3 -m tender_agent.cli status
```
