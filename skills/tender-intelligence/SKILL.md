# 标讯情报 Skill

需要联网时使用 `web-access`；处理 xlsx 时使用 Spreadsheets；本地执行入口为 `tender-agent`。

## 目标

把不同招投标网站中的公告转成统一、可追溯、可去重的标讯记录。默认范围是贵州省，默认行业是图文广告与园林绿化；两类共用信息源采集，但使用独立关键词和分类标记。

## 安全边界

- 信息源账号和密码只允许进入 `data/private/tenders.sqlite3` 或运行环境的密钥服务。
- 不在对话、日志、日报、代码、Skill 文档中输出明文密码。
- 登录网站前先确认该账号属于用户或用户所在组织，并遵守网站规则。
- 遇到验证码、短信验证、人脸验证或协议确认时，标记为“需要人工”，不要绕过。
- 日报必须保留公告原始网址；无法核实原文的候选项进入复核队列，不直接推送为确定信息。

## 调度边界

- 本 Skill 只定义标讯采集、筛选、发布规则，不负责创建或维持 Codex 定时任务。
- 不得自行创建“每日标讯本机增量更新”等 Codex 自动任务。
- 更新可由用户手动触发，或由用户另行配置的外部 Agent/自动化平台触发。

## 更新工作流

1. 读取 `config/regions.json` 中的当前区域。
2. 读取 `config/industries/` 中已启用行业及关键词。
   当前固定读取 `graphic-advertising.json` 和 `landscaping.json`，合并关键词后只采集一次，再为每条公告写入独立 `industry_categories` 和 `category_keyword_matches`。
3. 从数据库读取启用的信息源，优先处理无需登录的公开公告，再处理需要登录的平台。
4. 需要联网时加载 `web-access`，按照网站正常交互路径查询最近 24-48 小时公告。
5. 对每条候选公告提取统一字段，字段规范见 `references/tender-schema.md`。
6. 用公告规范化网址作为第一去重键；缺少网址时使用“项目名称 + 招标人 + 截止时间”指纹。
7. 区域明确命中排除省份时剔除；区域不明确时进入复核，不擅自判断为贵州。
8. 关键词只允许在以下字段匹配：项目名称；招标或采购内容；招标或采购范围；项目概况。项目名称命中任意一个关键词即收录；后三类内容字段命中任意一个关键词也收录。采购人、招标人、代理机构、资格条件、报名要求、公告媒介和其他正文内容中的关键词一律不计。
9. 网页“项目主要内容”只提取公告中明确标注为“招标内容”“采购内容”“招标范围”“采购范围”或“项目概况”的栏目；“采购需求”“项目主要内容”“简要技术要求”等相似栏目不能自动代入。没有指定栏目时留空并显示“公告未载明”，允许用户人工纠正。
10. 截止时间已过的公告不进入“今日可投”主列表，但可作为更正或结果公告保存。
11. 生成公开网页数据：新增数量、重点项目、最早截止项目、需人工复核项、来源异常。
12. 用户要求查询或运行标讯时，转入统一命令 `PYTHONPATH=src python3 -m tender_agent.unified_site update --publish`，不在本 Skill 停在单板块结果。

## 信息源维护

新增或更新信息源时：

- 将贵州省公共资源交易云的工程建设、政府采购、其他交易三个栏目同时作为图文广告和园林绿化的共享来源，分别使用 `https://ggzy.guizhou.gov.cn/xxfw/gcjs/`、`https://ggzy.guizhou.gov.cn/xxfw/zfcg/`、`https://ggzy.guizhou.gov.cn/xxfw/qttzjy/`；公开数据保留三个栏目各自的正式名称，不合并成笼统的“贵州省公共资源交易云”。
- 先更新用户维护的源表，再运行导入，不直接手改数据库。
- 一个网站可有多个公司账号，每个账号保留独立状态。
- 记录登录是否有效、最后验证时间、验证码类型、可查询地区、适配器状态。
- 网站结构变化时新增或修复对应适配器，不用脆弱的全站通用选择器替代。

## 新增来源开发 SOP

开发新网站采集器时，吸收 Scrapling 的抓取方法，但不把通用爬虫框架当成
业务判断来源。新增来源必须先完成来源结构判断，再写独立采集器和测试。

1. 先判断站点类型：优先查找公开 JSON/API；其次使用静态 HTML；只有静态层
   无法得到列表或详情时，才使用真实浏览器观察动态页面。
2. 对 SPA 或动态站点，先通过浏览器正常访问页面，观察 XHR/fetch 请求、分页
   参数、详情接口和必要 headers，再把稳定接口固化为采集器。
3. 调试新来源时保存少量列表页和详情页样本，用于单元测试和解析规则回归；
   不在调试中反复高频请求源站。
4. 选择器要有容错：除 CSS/XPath 外，同时参考标题文本、详情链接、发布时间、
   父级容器、栏目标签等多项特征；页面结构变化时优先修复该来源采集器。
5. 对正文交给 AI 或人工辅助判断前，先清理 script、style、注释、隐藏元素、
   不可见文本和无关导航，降低 prompt injection 和误读风险。
6. 只有在公开来源确实存在反爬或动态渲染阻碍、且符合网站规则和用户授权时，
   才允许使用浏览器抓取；遇到验证码、短信、人脸、协议确认或登录限制时，
   标记为需要人工，不绕过验证。
7. Scrapling、Playwright、CDP、MCP 等工具只作为新来源调研或实验工具；每日
   更新主链路必须保持可测试、可复现、可记录，不依赖 AI 会话式判断。
8. 每个新来源上线前必须补测试，覆盖列表解析、详情字段、关键词边界、区域
   过滤、已处理跳过、失败重试和来源正式名称。

## 行业维护

每个行业使用一个独立 JSON 文件，至少包含：

- `industry_id`
- `name`
- `keywords`
- `exclude_keywords`
- `match_mode`

同义词应加入关键词库。明显会产生大量误报的词应加入排除词，而不是在代码中临时判断。

用户明确确认的误报标题记录在 `config/excluded_notices.json`，每次生成公开数据时强制排除，防止旧数据或来源重复公告再次进入页面。

## 输出要求

统一标讯雷达的图文广告和园林绿化板块须显示施工粗筛的全部信息源，同时保留各自已有行业来源；两个板块都使用与施工粗筛一致的带数量信息源快捷按钮，点击按钮同步信息源下拉框，再次点击取消筛选。

给用户汇报时始终分开说明：

- 已采集并核实的标讯
- 疑似相关、等待人工复核的标讯
- 信息源登录或采集异常
- GitHub Actions 是否成功，以及 GitHub Pages 是否已更新

不要把“已列入信息源库”描述成“采集器已上线”。只有对应网站采集器、字段提取、关键词筛选、测试、GitHub Actions 和线上页面都验证成功后，才可称为上线。

当前已上线的独立采集器：

- 贵州省招标投标公共服务平台
- 黔云招采电子招标采购交易平台
- 遵义市公共交通（集团）有限责任公司
- 贵阳市公共资源交易国有企业招标采购平台
- 中烟电子采购平台
- 中国南方电网供应链统一服务平台
- 贵州省公共资源交易云：工程建设、政府采购、其他交易栏目
- 黔云招采子板块：贵州高速、贵州能源、黔晟国资、贵州水投
- 云农商电子招采平台
- “黔顺云采”集采平台
- 军队采购网贵州公告

## 本地入口

初始化附件：

```bash
PYTHONPATH=src python3 -m tender_agent.cli bootstrap \
  --sources /path/to/01信息源库.xlsx \
  --keywords /path/to/02图文广告行业关键词库.txt \
  --history /path/to/03标讯信息表.xlsx
```

查看数据状态：

```bash
PYTHONPATH=src python3 -m tender_agent.cli status
```
